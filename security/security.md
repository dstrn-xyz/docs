<a name="csrf-protection"></a>

## csrf protection

dframework automatically generates a cross site request forgery (csrf) token for each active user session managed by the application. this token is used to verify that the authenticated user is the one actually making the requests to the application.

the csrf token is cryptographically secure and relies on an hmac generated using your `APP_KEY`.

```javascript
// generate a new token
const token = req.csrfToken();

// verify a submitted token
if (req.verifyCsrfToken(submittedToken)) {
  // request is safe
}
```

<a name="rotating-tokens"></a>

### rotating tokens

csrf tokens generated by the framework are valid for an active window of six hours. if you need to manually rotate the token for security reasons, you can use the `rotateCsrfToken` method.

<a name="rate-limiting"></a>

## rate limiting

dframework includes a sliding window rate limiter designed to prevent abuse and brute force attacks. when a client exceeds the maximum number of allowed requests within the designated time window, the framework automatically responds with an http 429 "too many requests" status code.

<a name="configuration"></a>

### configuration

the rate limiter tracks requests seamlessly. if your application sits behind a load balancer or reverse proxy, the limiter correctly processes the `x-forwarded-for` headers based on your configured `trustedProxies` array, ensuring that rate limits apply to the actual client ip and not the load balancer itself.

the limiter automatically injects standard headers (`Retry-After`, `X-RateLimit-Limit`, `X-RateLimit-Remaining`) into the http response.

<a name="shield-guard"></a>

## shield guard

in addition to standard rate limiting, the framework features a bot mitigation engine named `ShieldGuard`.

`ShieldGuard` operates by generating a cryptographic challenge containing a timestamp, salt, and hmac signature. the client must successfully resolve a proof of work (pow) challenge based on the specified difficulty before they are allowed to proceed.

furthermore, the engine collects client side telemetry regarding the user's environment and behavioral signals (touch events, mouse velocity, window dimensions). these signals are strictly evaluated on the server side to compute a "bot score". if the score exceeds the acceptable threshold, the request is instantly rejected.

this transparent layer of defense ensures that automated scripts and malicious bots are halted before they ever reach your application controllers.
